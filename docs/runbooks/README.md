# docs/runbooks — Operational How-To

**Genre: RUNBOOKS** — step-by-step operational procedures (deploy, docker, testnet lifecycle, toolchain, prod images, environments). **Owners:** per the owning item's agent (implementation-plan Conventions, P-9): `env-inventory.md`/`deploy.md`/`deploy-komodo-cloudflare.md`/`environments.md` → robbed-architect · `toolchain.md`/`testnet.md` → robbed-contracts · `docker.md`/`prod-images.md` → robbed-indexer; architect ratifies. **Authority relation:** derived — a runbook never overrides the spec, a service doc, or a §12 decision; it transcribes them into operator steps. **One deliberate genre exception:** `env-inventory.md` is a CATALOG (machine-consumed contract, parsed by `scripts/env-sync-check.ts` + doc-check check g) housed here because its consumers and audience are operational — its path is frozen. Does **not** belong here: design rationale (→ `docs/services/`, `docs/architecture.md`) or decision records (→ `docs/decisions.md`). Implementation-plan verify clauses cite these exact paths — do not rename or move files here.
