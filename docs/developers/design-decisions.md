# Design decisions (binding record)

**What this is.** The append-only, dated record of every ratified protocol/product decision, the open items still in flight, and the project's go/no-go framing. It is **binding institutional memory** — the rulings here resolved real ambiguities, many by explicit user direction, and they govern the rest of the docs. When a decision here amends behavior, the *current* behavior is written into the owning doc ([contracts.md](contracts.md), [architecture.md](architecture.md), [threat-model.md](threat-model.md), [indexer.md](indexer.md), [api.md](api.md), [web.md](web.md), or the [users/](../users) pages); this record carries the dated provenance and reasoning.

**Authority.** The root authority is [README.md](../../README.md) + the developer docs under [docs/developers/](.). This record is part of that authority — it is not a tracker, plan, or status report; it is an ADR-style log of decisions that already landed. Entries are never deleted; a superseded decision is annotated in place with the entry that superseded it.

**How to read it.** Each ruling has a stable id `D-N` (dated; `USER-DIRECTED` marks the ones the user chose directly). Open items carry `O-*` / `OI-*` ids. Reasoning is kept where it is load-bearing; obsolete detail is compressed to "superseded by D-x". Cross-references use these ids and markdown links, never section numbers.

## Ratified decisions

### Foundational architecture (D-1 … D-10, 2026-07-09 unless noted)

- **D-1 — Graduation venue = Uniswap V3 (Option B).** Curve graduates to a Uniswap V3 1% full-range position; the LP NFT lives forever in an immutable `LPFeeVault` and the treasury collects trading fees. Supersedes the interview's V2+burn choice; **V2 + LP burn is the documented fallback only** (adopting it flips the LP copy to "LP burned forever" — the sole sanctioned flip).
- **D-2 — Wallets.** Classic wagmi/RainbowKit; ERC-4337 is a later platform-expansion item (see D-63).
- **D-3 — Fee custody.** Treasury (a Gnosis Safe) is the fee sink; a creator-fee leg was designed-in and initially disabled — **now live**, see D-63 / D-68 / D-69.
- **D-4 — Graduation threshold.** Originally sized to ~$69k mcap parity, constants fixed at deploy. **SUPERSEDED by D-67 (2026-07-13):** the target is now a **flat net-of-fee ETH raise** (5.7 ETH product default), env/override-driven at M0, not a USD-mcap derivation.
- **D-5 — Indexer.** Ponder → Postgres → Redis.
- **D-6 — Frontend.** Next.js on Bun, SSR + per-token OG images.
- **D-7 — Storage.** Cloudflare R2 + CDN; the integrity commitment is the on-chain `metadataHash`, not the mutable URL.
- **D-8 — Security posture.** AI-assisted audit pipeline **plus** a mandatory capped beta **plus** a public bounty before caps lift **plus** an explicit external-review decision gate. See [threat-model.md](threat-model.md) and [SECURITY.md](../SECURITY.md).
- **D-9 — Compiler.** One exact pin, no ranges (candidate `0.8.35`), validated against Blockscout before deploy (see O-5).
- **D-10 — Treasury contract.** Canonical Gnosis Safe (verify on-chain, else deploy the canonical contracts); bespoke multisig dropped.

### Curve, graduation & fee mechanics

- **D-11 — `GRADUATION_ETH` is net-of-fee real reserves** (the ETH that actually funds the LP). The final buy is clamped to land exactly on the threshold; the excess is refunded in the same tx.
- **D-12 — `ReadyToGraduate` locks BOTH buys and sells.** "Sells always open" means *no pause authority* exists — this lock is a deterministic, permissionlessly-exitable protocol state, not a pause. The UI shows a two-sided "Graduating…" interstitial.
- **D-13 — Graduation dust.** Token leg → `0x…dEaD` ("burned"; the token has no burn function); WETH leg → treasury. Real ETH value is never destroyed.
- **D-14 — Canonical LP copy.** One string everywhere: **"LP principal permanently locked; trading fees claimable by treasury."** — never "burned" (the V2-fallback flip is a separate branch, D-1). **Amended by D-69:** the mainnet creator-fee generation flips the fees clause to read "…split between treasury and creator" (principal still permanently locked; treasury-first ordering kept). The flip lands *with* that generation's deploy — the shared `LP_COPY` constant + copy-lint fixture and CLAUDE.md move at that time, never earlier — so the currently-enforced string stays "…claimable by treasury."
- **D-15 — Canonical event shapes** (indexer needs zero hot-path RPC reads): `TokenCreated(token, curve, creator, name, symbol, metadataHash, metadataUri, pool)` on the factory; `Trade(trader, isBuy, ethAmount gross, tokenAmount, fee, virtualEthReserves, virtualTokenReserves, realEthReserves — all post-trade)` on the curve. The creator's atomic initial buy is **not** in `TokenCreated` — it is derived from the first `Trade` in the same tx. Load-bearing ordering invariant: the factory emits `TokenCreated` **before** the initial-buy `Trade` (same-tx handler ordering is by `logIndex`).
- **D-16 — Holder-balance truth.** LaunchToken `Transfer` is the sole source of holder balances; pre-graduation V3 pool activity is **not** in the price series (the curve is the sole venue until `Graduated`). **Amended by D-75 (2026-07-14):** the V3-leg trade attribution (`trader`) and the best-effort V3 cost-basis `holder` now derive from the transaction-sender EOA (`event.transaction.from`), NOT `Swap.recipient` — closes indexer.md OI-5. The `Transfer`-is-sole-balance-truth core and the pre-grad-not-indexed ruling here are UNCHANGED; only the recipient-keyed V3 attribution is superseded.
- **D-17 — Candle intervals:** 1s · 15s · 1m · 5m · 15m · 1h.
- **D-18 — Anti-sniper mechanism = `block.timestamp` window** (never `block.number`, never `arbBlockNumber`); seconds are deployment-constant while block cadence is a marketing figure. Window/cap values from M0 (D-32).
- **D-25 — Curve fee escrow = pull-payment (load-bearing).** `BondingCurve` **never** transfers ETH to the treasury on any trade path; the fee accrues in-contract (`accruedFees`) and is withdrawn by a separate **permissionless, non-phase-gated `sweepFees()`**. This makes "sells always open" true *by construction* — a hostile/reverting treasury cannot freeze buys or sells. Solvency invariant becomes `balance ≥ realEthReserves + accruedFees`; exact-fee invariant becomes `treasuryReceipts + accruedFees == Σ fees`. Resolves threat UM-1.
- **D-26 — Graduation fee = small flat, cost-based** (≈ V3-migration gas + a thin margin), **not** a %-of-raise. Exact number finalized at M1/deploy against real gas; M0 carries it as a gas-derived formula, never a hardcoded USD figure.
- **D-27 — Anti-sniper v1 frozen; redesign deferred.** v1 ships the fixed `block.timestamp` window + per-tx `MAX_EARLY_BUY` cap (values from M0). A decaying + size-based early-buy fee is a pre-caps-lift roadmap item, not an M1 change.
- **D-33 — Arb-back parameters + budget rule.** Values (D-33): `toleranceTicks = 100`, `maxArbIterations = 8`, `migrationSlippageBps = 100`. The migrator's arb-back spends at most `MIGRATION_SLIPPAGE_BPS` of *each leg's own mint requirement*, symmetric per leg — token leg may draw down to `LP_TOKEN_TRANCHE·(1−bps)`, WETH leg may spend `wethForMint·bps` — with the mint's `amount0Min`/`amount1Min` re-enforcing the parity floor independently. Proven in gate-2 invariant 6. Residual (mispricing beyond the recoverable range) is the UM-2 caps-lift item, see D-61. Mechanics in [contracts.md](contracts.md).
- **D-34 — Graduation caller reward.** M0 default with an immutable 5× hard ceiling; re-validated ≥10× actual `graduate()` gas at M1 (fork-measured). Retunable in the capped beta.
- **D-61 — Gates 5 + 6 executed on a live mainnet-fork (UM-2 grief-lock dispositioned).** Safety verdict PASS across 384 sampled lifecycles (no fund-loss; invariant 6 held). The graduation grief-lock is **uneconomic and self-correcting**: sustaining it costs the attacker ~0.10–0.20 ETH at zero profit, is non-permanent, and a harmed holder can profitably correct the pool so the next `graduate()` succeeds. Sanctioned closure (neither needs user ratification): (i) a permissionless **correct-and-graduate** periphery contract, and (ii) gate-7 stall monitoring. The **only** user-decision residual is OPTIONAL: a sells-reopening timeout hatch that would touch the D-12 two-way lock — **recommendation on file is to ship without it.** Tuning lever: `MIGRATION_SLIPPAGE_BPS` widens attacker cost (calibration-safe); `TOLERANCE_TICKS` is BLOCKED by the `GradCalibrationGuard` bound. Gate-6 also fixed the trade-UI default slippage floor at 2% (`DEFAULT_SLIPPAGE_BPS = 200`).

### Anti-sniper & economics values (M0 defaults, tunable in the capped beta)

- **D-32 — Anti-sniper values.** `antiSniper.windowSeconds = 8` (inside the 5–10s band) and `antiSniper.maxEarlyBuyWei = 2.5% × GRADUATION_ETH`. M0 engineering defaults, retunable during gate 7; multi-wallet bypass remains acknowledged (see threat-model attacker catalog).
- **D-35 — Large-value confirmation-disclosure ETH basis furnished at M0; threshold decided in D-47.**
- **D-36 — M0 governance values computed.** `governance.organicVolumeFloor` (Gate G-A.1 market floor, = 5 × `GRADUATION_ETH` per 7d of **own-indexer** organic volume — NOT a headline metric) is a PRODUCT/POLICY default **requiring architect + effectively user ratification before Gate G-A** and recalibration at M2. `governance.clusterAlertThresholds` (gate-7 funder-cluster alert: per-token 25% / platform 10% over 24h) are advisory-only engineering defaults, tuned with robbed-security before the beta deploy. Values live in `constants.json.governance`, never inlined.
- **D-47 — Large-value confirmation threshold DECIDED = 1.0 ETH.** A trade whose ETH notional is ≥ 1.0 ETH triggers the stronger posted-to-L1 / finalized disclosure UX; below it, the default treatment applies. Config (`NEXT_PUBLIC_LARGE_VALUE_ETH_THRESHOLD` + an API mirror), never a code literal. Retunable in the capped beta.
- **D-62 — M0 economics re-derived on a mainnet-fork (2026-07-12).** Same reviewed methodology; two independent drivers moved the numbers — a **durable** gas-model fix (fork-measured `graduate()`+V3 migration ≈ 817,845 gas worst-case) and an ETH/USD **snapshot** re-peg. Hard rule for deploy: the ETH-pegged curve constants are a snapshot valid only at that ETH/USD; they **MUST be re-derived and re-locked at mainnet-deploy time against the then-current sourced ETH/USD** — the deploy runbook gates on a fresh derivation immediately before the mainnet tx. `GRADUATION_ETH` later dropped out of the re-peg set when D-67 made it a flat ETH input. `constants.json` is the single source of truth; values are never inlined.
- **D-67 — USER-DIRECTED: `GRADUATION_ETH` becomes a flat ETH input** (product default **5.7 ETH**, tick-aligned by M0 to the current `curve.graduationEthWei`), abandoning the $69k-mcap premise (D-4). Net-of-fee semantics (D-11), graduation mechanics, and the D-12 lock are unchanged. Set ETH-to-ETH against RobinFun's ~5.74 ETH graduation bar (user benchmark, 2026-07-13) — **no USD figure enters code**. Because the bar is a flat ETH constant, `GRADUATION_ETH` drops out of the D-62 deploy-time re-peg set; `CREATION_FEE` (~$1.50-equiv) still re-pegs. Self-referential multiples rescale automatically: `MAX_EARLY_BUY = 2.5% × GRADUATION_ETH`, `organicVolumeFloor = 5 × GRADUATION_ETH`; `largeValueEthThreshold` stays a fixed 1.0 ETH config. Testnet (46630) MAY keep a small faucet-scale `GRADUATION_ETH` (the 0.05 ETH/24h faucet can't fund a 5.7 ETH raise); 5.7 ETH is the mainnet/product target only.

### Creator fees (live feature — the final-version fold)

- **D-63 — USER-DIRECTED scope change: Phase 2 folded into the final version, SEQUENCED.** Of the four folded items, **only creator fees ships** — comments are removed (D-64), and ERC-4337 + livestreams remain out (post-caps-lift M6, each needs its own mini-spec). Creator fees fold into the pre-mainnet contract surface (M4) as a **fresh immutable factory generation** (new `CurveFactory` + `BondingCurve` + `Router` + pull-payment `CreatorVault`) — not an upgrade; existing v1 curves keep `creatorFeeBps = 0` forever. This REOPENS security gates 1–6 + the gate-9 decision for the changed surface; the D-25 pull-payment property and gate-2 exact-fee/solvency invariants extend to the creator leg. **Split model (RATIFIED): additive, not a carve-out** — `totalTradeFeeBps = treasuryFeeBps + creatorFeeBps`, constructor-asserted `≤ MAX_TRADE_FEE_BPS = 200` (the ≤2% hard cap), computed in-contract, never caller-supplied.
- **D-64 — USER-DIRECTED scope removal: per-token comments REMOVED from scope** (reverts the D-63 comments leg). The SIWE-authored, moderation-gated comments panel + its table / endpoints / WS channel / rate-limit / user-facing SIWE are deleted end-to-end. **Admin-SIWE and listing moderation are unaffected.** De-scoped, not deferred — a future reintroduction is a new additive flow. Creator fees (D-63) are unaffected.
- **D-68 — USER-DIRECTED: mainnet creator fee ON at 50 bps (0.5%).** Sets the D-63 knob: `creatorFeeBps = 50`, additive with `treasuryFeeBps = 100` → `totalTradeFeeBps = 150` (1.5%) ≤ 200. This is the **pre-graduation (curve) trade-fee split**. Supersedes the "hardcoded 0" mainnet framing of the original creator-rewards design. The M0 re-derivation obligation still stands (fold the additive 50-bps leg into curve calibration for the D-67 target). The 50-bps creator fee **accrues in-contract and is pull-withdrawn** via `CreatorVault`, never pushed on a trade path — a hostile/reverting creator cannot freeze buys or sells (`balance ≥ realEthReserves + accruedFees + accruedCreatorFees`). The `== 200` boundary is exercised only by gate-2 tests, never the deploy default.
- **D-69 — USER-DIRECTED + ARCHITECT-DESIGNED: post-graduation V3 fees SPLIT 50/50 treasury/creator at `collect()`; LP PRINCIPAL STAYS PERMANENTLY LOCKED.** The creator-aware `LPFeeVault` (part of the D-63 generation) harvests both token legs to itself, splits each 50/50 (`creatorLpShareBps = 5000`), pushes the treasury half to the fixed immutable treasury and the creator half to the pull-payment `CreatorVault`. The migrator registers `tokenId → creator` at graduation (migrator-gated, set-once) so `collect()` needs zero external identity calls. **Why 50/50:** it makes the creator's absolute rate venue-invariant — 0.5% of volume on the curve (D-68) *and* 0.5% of volume post-graduation — one honest "0.5% of your token's lifetime volume" story with no discontinuity at graduation, matching RobinFun and beating the incumbent's 0% post-grad. **Principal-lock guarantee is unchanged** — the vault never calls `decreaseLiquidity`, holds the NFT forever, no owner / no withdraw. `collect()` stays permissionless and un-brickable (the creator share routes through *our* non-reverting `CreatorVault`). **Amends the D-14 canonical LP copy** — the fees clause becomes "…split between treasury and creator" **with this generation's deploy** (mainnet ships only this generation), never flipped earlier. **Requires a security re-gate** (folds into the D-63 M4 re-gate); gate-2 invariants: per-leg split sums exactly to collected, principal liquidity monotonic non-decreasing, hostile creator/treasury cannot brick `collect()`, `creatorOf` set-once/unspoofable. Mechanics in [contracts.md](contracts.md). Docs-first basis: Uniswap NPM `collect` harvests fees in both tokens without touching liquidity.

### Confirmation tiers & indexer materialization

- **D-20 — Confirmation upgrades propagate as O(1) watermark broadcasts** on `global:confirmations` (clients upgrade held events locally); REST serves the materialized per-row state; no per-row fanout.
- **D-48 — M2-start environment checks.** (a) ETH/USD source is config-driven — Chainlink-if-present, else DefiLlama/Coinbase HTTP with source+timestamp (Chainlink CONFIRMED on 4663, see D-51). (b) `safe`/`finalized` RPC block-tags are SUPPORTED on the official 4663 RPC (the tag path runs; the L1-watermark reader stays a dormant fallback). (c) Ponder does **not** tolerate external `UPDATE` of a hot `confirmation_state` column (its indexing-store cache re-flushes and reverts it) — so the tier is **read-derived**: a `confirmation_watermarks` singleton + a shared `stateForBlock` rule computes a row's tier at read time as a pure function of `block_number` vs the watermarks. Monotonicity holds structurally; no per-row join table. Satisfies D-20 unchanged.
- **D-51 — Chainlink ETH/USD feed CONFIRMED on 4663** (proxy `0x78F3556b67E17Df817D51Ef5a990cDaF09E8d3A9`, aggregator `0x6091E64eb7138EEF066a80FD3A0d7427B91f2721`; live-verified `description()=="ETH / USD"`, `decimals()==8`). Chainlink branch selected for the indexer's snapshot poller with **mandatory fail-closed startup assertions**. Mainnet-only; LOCAL/TESTNET use the DefiLlama/Coinbase HTTP fallback.
- **D-56 — USER-DIRECTED UX carve-out: the "Soft-confirmed" chip + its L2-finality tooltip are removed from the trade UI.** Scope is narrow (a): only the tier-(1) *visible badge* + tooltip go (trade widget, live trade feed, launch labeling). **KEPT and still binding:** all three-tier machinery — indexer tracking, the never-final-while-soft reconcile, txHash reconciliation, the D-20 watermark broadcast, and **posted-to-L1 / finalized surfacing** (incl. the D-47 large-value ≥1.0 ETH escalation). With no soft chip, a fresh trade simply shows no finality claim rather than an optimistic-but-labeled one; it can never show a stronger tier than the watermark permits. This NARROWS, not repeals, "confirmation tiers everywhere in UX" for tier (1)'s badge only. The dead `awaitingIndex` badge-tooltip affordance is intentionally gone; the WS-silence *behavior* (keep the row + REST-poll) is retained.
- **D-75 — USER-DIRECTED: V3 (post-graduation) trade attribution keys on the transaction-sender EOA, not `Swap.recipient`.** *(2026-07-14; owner robbed-indexer; indexer-materialization change — no protocol / contract / shared-shape change; requires a reindex.)* For a V3 `Swap` on a graduated pool, the `trades.trader` column **and** the best-effort V3-leg cost-basis `holder` (`total_eth_in/out` attribution) derive from **`event.transaction.from`** — the EOA that submitted the swap — not from `Swap.recipient`. **Supersedes the V3-attribution clause of D-16 / closes indexer.md OI-5** (D-16's `Transfer`-is-sole-balance-truth and pre-grad-not-indexed rulings are UNCHANGED — only the recipient-keyed mapping is replaced). **Rationale:** for a V3 sell that outputs ETH, `Swap.recipient` is SwapRouter02 (it receives WETH to unwrap into ETH for the user), so recipient-keying showed the router as the "seller" in the Token Detail Trades table; `event.transaction.from` is the user's EOA for a direct SwapRouter02 swap — correct for both buys and sells — and makes per-EOA V3 cost basis accurate for directly-submitted swaps. **Caveat (same best-effort class as before):** a contract-mediated / aggregator-relayed swap attributes to the relayer (`tx.from` = relayer, not the end-user), so V3 cost basis stays labeled **best-effort** and the D-16-core exact balances (via `Transfer`) are untouched; tightening this is out of scope until the Phase-2 4337 work (D-63). **Operational consequence:** this changes the indexer swap-handler derivation ⇒ a Ponder handler `build_id` change ⇒ a **full reindex is required** on deploy. No copy change (LP / confirmation-tier copy untouched, `.claude/rules/lp-copy.md`); no market-metric literal enters (`.claude/rules/no-market-metrics.md`). Mechanics in [indexer.md](indexer.md) §3.4 / §3.6.

### Frontend surface (the four-page redesign)

- **D-50 — USER-DIRECTED four-page redesign (authoritative).** v1 ships **four** pages, not three: **Portfolio `/portfolio`** (read-only, no new tx types, no `collect()` UI) is pulled from Phase 2 into scope, and `/launch` → **`/create`**. Brand `ROBBED_`; a terminal-mono skin over shadcn primitives; mobile-first (dark-first stands). **(f) Discover surface (USER-RATIFIED):** the shipped `/` is a single terminal panel — a **TRENDING carousel** (top tokens by 24h volume, API-ordered) over the **live event tape**. The King-of-the-Hill hero, token grid, its sorts/filters, and grid URL-state are **retired from the page** but remain **API capabilities** (`GET /v1/tokens` sort/filter + the ranking formulas, D-22); header search stays. All data/protocol rules are unchanged. **(f) Amended by D-70 (2026-07-14):** the token-card **grid is re-added** to Discover below the carousel + tape (the retired grid's `GET /v1/tokens` capability regains a web consumer); sorts/filters return as a **view-local** control (grid URL-state stays retired — only `?q=` remains a URL param). The carousel + tape are unchanged.
- **D-70 — USER-DIRECTED: Discover re-adds a rich per-token card grid (pump.fun / robinhood.fun style), and per-token card metrics live-update over WS.** *(2026-07-14; amends D-50(f), which retired the grid FROM THE PAGE while keeping it an API capability — D-70 re-attaches the web consumer, it does not un-retire a protocol capability.)*
  - **Surface.** The TRENDING carousel + live event tape (D-50(f)) **stay**; a responsive **TokenCard grid** (1/2/3 columns) renders below them as the primary browse surface, hydrated from SSR `GET /v1/tokens` and cursor-paginated ("load more", page size 48). Header search (D-4/DISC-4) unchanged.
  - **Card fields (exact).** image · name · ticker · **description** (NEW — a card-preview blurb) · creator · age · **mcap** · **Vol 24h** · **24h Δ%** · **graduation status**. Every metric is an **indexer aggregate rendered by reference — never client price math, never a constant** (no-market-metrics; carries DISC-1/DISC-3's discipline onto the grid).
  - **Denomination (no-market-metrics).** Testnet has **no live ETH/USD feed** (D-51), so cards render **ETH-denominated**: mcap from `mcapEth` (wei), Vol 24h from `volume24h` (wei). A **USD mirror renders only where the live `GET /v1/meta/eth-usd` feed exists** (the card's `mcap` UsdValue, computed server-side / derived as `mcapEth × ethUsd`); **no fabricated USD** on testnet.
  - **Graduation-status copy (HARD RULE — D-14 / D-65 / `.claude/rules/lp-copy.md`).** Curve → **"{n}% to graduation"** (`n` from the card's `progressPct`, formatted via the shared `GraduationProgress` `pctText`). Graduated → **"Graduated · Uniswap V3"** (venue named; reuses the codebase's existing "→ Uniswap V3" phrasing). The card **NEVER** says "burned" / "LP burned" 🔥. The full LP-destiny sentence (the `LP_COPY` constant) stays **token-detail-only** (D-65 carve-out) — the card carries only the venue/status label, not the LP sentence.
  - **Sorts / filters (view-local, not URL — D-50).** The API already exposes `sort ∈ {trending,newest,mcap,volume24h,progress}` (D-22) and `filter ∈ {pregrad,graduated,all}`. The grid re-adds a **minimal view-local control**: default `sort=trending`; sort tabs {trending, newest, mcap, volume24h}; filter tabs {all, pregrad, graduated}. Rendering is **server-authoritative** — the client renders the API's returned order verbatim, never re-ranks.
  - **Live metrics over WS (the freshness fix).** Symptom: the tape appends new trade EVENT rows live, but the aggregate registry the cards/rows read by reference is **not patched** on those events, so mcap / Vol 24h / Δ% / progress go stale until reload. Mechanism (chosen): on each **indexed trade** and on **graduation**, the indexer publishes a **coalesced per-token `token_metrics` snapshot** — the authoritative recomputed ETH aggregates `{ priceEth, mcapEth, volume24h, change24hPct, progressPct, status, graduated }` — on a **new global channel `global:metrics`**, throttled per token (≤1 / `WS_METRICS_THROTTLE_MS` ≈ 2 s, trailing-edge, last-write-wins by `blockNumber`). The frontend patches the cached `tokens` list + tape registry **by reference** via TanStack Query `setQueryData` (functional immutable updater, docs-verified 2026-07-14) — **no refetch, no client math**. **Rejected alternatives:** (a) deriving aggregates on the client from the `trade` WS payload — forbidden (no-market-metrics; a single trade can't justify an aggregate; the payload lacks `volume24h`/`progressPct` anyway); (b) a throttled full-list refetch per trade (the `use-live-token` precedent) — hammers the API and reorders the grid. REST stays source of truth (WS is a patch stream); on WS reconnect / seq-gap, ERR-11's `tokens`-family invalidation rebuilds the registry.
  - **Data gap (precise).** The **only** new card field is `description` — already an indexer column (`tokens.description` / verifier `metadata_verifications.description`), already SELECTed by `TOKEN_LIST_SELECT` and mapped into every `TokenListRow` — so it needs **no indexer change**: only `tokenCardSchema.description` (shared) + `toTokenCard` projection (API, truncated to `TOKEN_CARD_DESCRIPTION_MAX`). All other fields (`mcap`/`mcapEth`, `creator`, `volume24h`, `change24hPct`, `progressPct`, `status`, `graduated`) are already on the card.
  - **Ownership.** Shared shapes → **robbed-shared** (`tokenCardSchema.description`, `TOKEN_CARD_DESCRIPTION_MAX`, `wsTokenMetricsDataSchema` + the `token_metrics` union member, `GLOBAL_METRICS` + its entry in `GLOBAL_CHANNELS`); API projection + indexer publish point → **robbed-indexer** (docs in [api.md](api.md) / [indexer.md](indexer.md)); grid UI + WS subscribe/patch → **robbed-frontend** ([web.md](web.md) §3.1). `apps/web/e2e/user-flows.md` gains a **new `DISC-5`** (grid render + live-metric patch on a trade) → **robbed-e2e** (stable-ID rule — extend `DISC-*`, do not un-retire the tombstoned grid flows).

## Project framing & go/no-go gates

The project runs in **two phases with a decision gate between them.**

**Phase A — portfolio-grade testnet build (committed, timeboxed).** Scope: M0 → M3 fully on Robinhood Chain **testnet** (46630), public repo from day 1, plus one technical write-up. Hard timebox: ~3 weeks of part-time effort. **No legal work required in Phase A** (no mainnet, no fees collected). Exit artifacts stand alone as portfolio value regardless of the Phase B decision: verified contracts + invariant suite, live indexer with bot-detection, the four-page frontend, and the write-up.

**Gate G-A — mainnet go/no-go (end of Phase A).** Proceed to Phase B only if **ALL** hold:
1. **Market** — Robinhood Chain memecoin activity has not collapsed, measured against the **own-indexer** organic-volume floor (not headline metrics; the floor is the open item above).
2. **Competition** — the incumbent has not consolidated the niche (still < order-of-magnitude ahead on daily token creations, or visibly failing).
3. **Personal** — income security resolved or on a clear path; mainnet operation does not compete with it.
4. **Legal** — MiCA/JDG consultation completed with a workable path (the legal-wrapper open item moves from "open" to **blocking** here).

**Phase B — capped mainnet launch (conditional).** Only after G-A passes: M4 (security gates 5–8), the capped beta, the public bounty, then caps lift (M5). **If G-A fails:** the repo is archived as a portfolio artifact, the write-up is published, loss = zero beyond the timeboxed effort.

**Milestones.** M0 parameter notebook → M1 contracts + gates 1–4 (testnet deploy) → M2 indexer/API → M3 frontend → M4 creator-fee factory generation + gates 5–8 + capped beta + bounty → M5 caps lift (gate-9 decision executed; gate-10 known-risks published) → M6 platform expansion (ERC-4337, livestreams — post-caps-lift, each its own mini-spec). All ten security gates are required before caps lift; the capped beta is mandatory.
