## Goal (end state)

> **This Goal == spec ¬ß14 Phase A (v1.2).** The two-phase framing added in spec ¬ß14 maps exactly onto this plan: **Phase A** = this Goal (M0‚ÜíM3 on testnet + verified contracts + bot-detecting indexer + 3-page frontend + write-up, "production-ready, not production-launched"); **Gate G-A** = the mainnet go/no-go at Phase-A exit (market/competition/personal/legal ‚Äî ¬ß14, all NEEDS-USER); **Phase B** = M4/M5 (gates 5‚Äì8, capped beta, bounty, caps lift), which begins only if G-A passes. ¬ß14 does not expand this Goal ‚Äî it names the structure the Goal already implied and makes the mainnet decision explicit and conditional. Nothing before Gate G-A depends on a G-A input.

ROBBED_ is **production-ready, not production-launched**: the entire stack (six contracts, indexer, API+WS, three-page frontend) is implemented as a transcription of the ratified service docs, runs locally with one command including seeded demo data, passes every automated test tier (Foundry unit/fuzz/invariant/fork = security gates 1‚Äì4, Vitest units, and a **Playwright e2e harness covering every user flow in the flow catalog** `docs/user-flows.md` ‚Äî the goal is verified, not assumed: every transaction type and every ¬ß5 user flow demonstrably works locally end-to-end through the real stack, asserted at all three layers: on-chain state, indexed record, reconciled UI), is green in CI on all jobs, is deployed and Blockscout-verified on Robinhood Chain testnet with the full token lifecycle (create ‚Üí trade ‚Üí graduate ‚Üí collect) exercised there, and has a prepared ‚Äî not executed ‚Äî production deployment: env-var inventory, deploy runbook (contracts + Safe handover + hosting), and buildable infra configs. **Mainnet launch, the capped beta (gate 7), the bug bounty (gate 8), and gates 5‚Äì10 generally are explicitly outside this goal** ‚Äî they are the M4/M5 follow-up (spec ¬ß10, ¬ß11) and several are blocked on human decisions (¬ß13: Safe signers, beta caps, moderation vendor, bounty terms).

## Conventions

- **Item format:** `- [ ] ID (owner) ‚Äî what. Transcribes: <doc ¬ß>. Verify: <command / file check>.` The verification command is the contract: `/goal` re-runs it before trusting a `[x]`.
- **Marking done:** `/goal` flips `[ ]` ‚Üí `[x]`, appending `‚Äî done YYYY-MM-DD; evidence: <one-line command result or commit>`. A failed re-verification re-opens the box (strike the evidence, note why).
- **Every item implicitly includes** the development-flow.md ¬ß5 loop: docs-first (context7/WebFetch for every library touched), service-doc coverage before code, tests per the doc's testing section, self-check greps, `/spec-check` on the diff, and ‚Äî for any `contracts/` diff ‚Äî the hoodpad-security adversarial gate. These are not repeated per item. **The mandatory `/spec-check` is the LOCAL command; CI's `spec-check` job skipping when `ANTHROPIC_API_KEY` is absent (e.g. on forks) is NOT a pass ‚Äî a skipped CI spec-check never satisfies the gate (P-8).**
- **Path-ownership extension (ratified here by hoodpad-architect, process-level):** `tools/localstack/**` (compose, orchestration, health) ‚Üí **hoodpad-indexer** (its services are the center of the stack); `tools/localstack/seed-chain*` + anvil/deploy glue ‚Üí **hoodpad-contracts**. Root `package.json` scripts wiring these ‚Üí whoever lands the item, architect sign-off. Everything else per development-flow.md ¬ß4.
- **Ownership ratifications (P-9, 2026-07-09):** root **`docker-compose.yml`** ‚Üí **hoodpad-indexer** (M2-0). **`docs/runbooks/**`** ‚Üí **authored by the owning item's agent** (env-inventory/deploy ‚Üí hoodpad-architect; toolchain/testnet-lifecycle ‚Üí hoodpad-contracts; monitoring/infra ‚Üí hoodpad-indexer), architect ratifies ‚Äî *not* exclusively architect. **T-2** (testnet Safe) ‚Üí **hoodpad-architect + ops** (canonical Safe only, ¬ß6.6). **`docs/user-flows.md`** and **`docs/user-flows-waivers.md`** ‚Üí **authored by hoodpad-frontend, ratified by hoodpad-architect** ‚Äî a sanctioned exception to "docs/*.md root = architect-owned" (development-flow ¬ß4), because the frontend owns the flow surface; the catalog's authority position is recorded in development-flow.md ¬ß1. **`packages/*` + workspace config** ‚Üí **hoodpad-shared** (development-flow ¬ß4).
- **Ambiguities:** never self-resolved (development-flow ¬ß3). Items marked **NEEDS-USER** stop the `/goal` loop and are surfaced.
