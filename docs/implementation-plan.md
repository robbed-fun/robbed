# hoodpad — Implementation Plan (master meta-plan)

**Status:** v1.0, 2026-07-09. Owned by hoodpad-architect; executed by the `/goal` command (`.claude/commands/goal.md`) as a goal-driven loop.
**Authority:** this plan is a *sequencing* of work already specified elsewhere — every item transcribes a section of `launchpad-spec.md`, `CLAUDE.md`, or a `docs/services/*.md`. Nothing here overrides spec §12; items blocked on spec §13 say so inline with the owner. If this plan and the spec/service docs disagree, they win and this plan gets fixed (development-flow.md §1).

## Goal (end state)

> **This Goal == spec §14 Phase A (v1.2).** The two-phase framing added in spec §14 maps exactly onto this plan: **Phase A** = this Goal (M0→M3 on testnet + verified contracts + bot-detecting indexer + 3-page frontend + write-up, "production-ready, not production-launched"); **Gate G-A** = the mainnet go/no-go at Phase-A exit (market/competition/personal/legal — §14, all NEEDS-USER); **Phase B** = M4/M5 (gates 5–8, capped beta, bounty, caps lift), which begins only if G-A passes. §14 does not expand this Goal — it names the structure the Goal already implied and makes the mainnet decision explicit and conditional. Nothing before Gate G-A depends on a G-A input.

hoodpad is **production-ready, not production-launched**: the entire stack (six contracts, indexer, API+WS, three-page frontend) is implemented as a transcription of the ratified service docs, runs locally with one command including seeded demo data, passes every automated test tier (Foundry unit/fuzz/invariant/fork = security gates 1–4, Vitest units, and a **Playwright e2e harness covering every user flow in the flow catalog** `docs/user-flows.md` — the goal is verified, not assumed: every transaction type and every §5 user flow demonstrably works locally end-to-end through the real stack, asserted at all three layers: on-chain state, indexed record, reconciled UI), is green in CI on all jobs, is deployed and Blockscout-verified on Robinhood Chain testnet with the full token lifecycle (create → trade → graduate → collect) exercised there, and has a prepared — not executed — production deployment: env-var inventory, deploy runbook (contracts + Safe handover + hosting), and buildable infra configs. **Mainnet launch, the capped beta (gate 7), the bug bounty (gate 8), and gates 5–10 generally are explicitly outside this goal** — they are the M4/M5 follow-up (spec §10, §11) and several are blocked on human decisions (§13: Safe signers, beta caps, moderation vendor, bounty terms).

## Conventions

- **Item format:** `- [ ] ID (owner) — what. Transcribes: <doc §>. Verify: <command / file check>.` The verification command is the contract: `/goal` re-runs it before trusting a `[x]`.
- **Marking done:** `/goal` flips `[ ]` → `[x]`, appending `— done YYYY-MM-DD; evidence: <one-line command result or commit>`. A failed re-verification re-opens the box (strike the evidence, note why).
- **Every item implicitly includes** the development-flow.md §5 loop: docs-first (context7/WebFetch for every library touched), service-doc coverage before code, tests per the doc's testing section, self-check greps, `/spec-check` on the diff, and — for any `contracts/` diff — the hoodpad-security adversarial gate. These are not repeated per item. **The mandatory `/spec-check` is the LOCAL command; CI's `spec-check` job skipping when `ANTHROPIC_API_KEY` is absent (e.g. on forks) is NOT a pass — a skipped CI spec-check never satisfies the gate (P-8).**
- **Path-ownership extension (ratified here by hoodpad-architect, process-level):** `tools/localstack/**` (compose, orchestration, health) → **hoodpad-indexer** (its services are the center of the stack); `tools/localstack/seed-chain*` + anvil/deploy glue → **hoodpad-contracts**. Root `package.json` scripts wiring these → whoever lands the item, architect sign-off. Everything else per development-flow.md §4.
- **Ownership ratifications (P-9, 2026-07-09):** root **`docker-compose.yml`** → **hoodpad-indexer** (M2-0). **`docs/runbooks/**`** → **authored by the owning item's agent** (env-inventory/deploy → hoodpad-architect; toolchain/testnet-lifecycle → hoodpad-contracts; monitoring/infra → hoodpad-indexer), architect ratifies — *not* exclusively architect. **T-2** (testnet Safe) → **hoodpad-architect + ops** (canonical Safe only, §6.6). **`docs/user-flows.md`** and **`docs/user-flows-waivers.md`** → **authored by hoodpad-frontend, ratified by hoodpad-architect** — a sanctioned exception to "docs/*.md root = architect-owned" (development-flow §4), because the frontend owns the flow surface; the catalog's authority position is recorded in development-flow.md §1. **`packages/*` + workspace config** → **hoodpad-shared** (development-flow §4).
- **Ambiguities:** never self-resolved (development-flow §3). Items marked **NEEDS-USER** stop the `/goal` loop and are surfaced.
